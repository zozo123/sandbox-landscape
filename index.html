Thesis You are not buying a sandbox. You are buying execution guarantees.
  • How dangerous is the code?
  • How long does it run?
  • Does state need to survive?
  • Does it need CI, GPU, RL, or orchestration?
  • Can you capture logs, diffs, artifacts, and scores?

Use cases

Start here.

Most mistakes come from using a short-lived code interpreter where a stateful agent computer, broker, or durable workflow is actually needed.

01

Small code and data analysis

Use model-native code execution, WASM, E2B, or Modal when state is minimal and the task is short.

02

Agent evals and benchmarks

Use Harbor plus Modal, Daytona, E2B, or Docker depending on scale, repo realism, and isolation needs.

03

GitHub Actions and PR validation

Use GitHub-hosted runners first, ARC for private autoscaling, and isolated sandboxes for AI-generated work.

04

Long-running coding agents

Use Daytona, Runloop, Islo, Tensorlake, or exe.dev when installed state, caches, logs, and partial work must survive.

05

RL rollouts and auto-spawning

Use Modal and Harbor for rollout scale. Add Daytona, Tensorlake, or Crabbox when environments need state or provider routing.

06

Production-adjacent automation

Use hardened stateful environments, credential boundaries, network policy, audit logs, and human approval gates.

Method

How this should evolve.

Public information only

Use public docs, public repos, public product pages, changelogs, and reproducible experiments.

Agent disclosure

Every benchmark should disclose prompt, agent/model, date, commit, harness, raw logs, and human edits.

Evidence over positioning

Vendor claims are leads. Benchmark results need code, logs, artifacts, scoring rubrics, and failure notes.

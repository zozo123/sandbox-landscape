Decision rule

You are buying execution guarantees.

  1. How dangerous is the code?
  2. How long does it run?
  3. Does state need to survive?
  4. Does it need GitHub or CI integration?
  5. Does it need GPU, RL, or post-training scale?
  6. Does it need auto-healing, retries, or orchestration?
  7. Does it need enterprise credential control?

Reference architecture

Use a sandbox portfolio, not one backend.

Put a control plane above the providers so evals, CI, long-running agents, and RL rollouts can evolve independently.

Model layer OpenAI / Anthropic / Gemini
Agent loop LangGraph / OpenAI Agents SDK / custom agent
Durability Temporal or equivalent workflow engine
Router Sandbox broker selects backend by guarantees: Crabbox, custom broker, or provider router

Eval / RL

Harbor + Modal for benchmark harnesses, reward execution, and parallel rollouts.

Stateful work

Daytona, Runloop, Islo, exe.dev, or Tensorlake for long-lived repo and browser tasks.

CI safety

GitHub Actions ARC, Docker Sandboxes, and isolated runners for PR validation.

Survival

Durable orchestration

Temporal, Inngest, Trigger.dev, Dagster, Prefect, Argo Workflows, and Kubernetes-native sandbox APIs.

Benchmark transparency

Future benchmarks must be reproducible and attributed.

This repo should grow gradually. Every benchmark should be created from public information, cite its sources, and disclose the agent used to create or run it.

Public-info only

Use public docs, public repos, public product pages, changelogs, and reproducible local experiments.

Agent disclosure

Record the agent, model, prompt, date, repo commit, tools, and human edits for each benchmark.

Evidence over claims

Publish harness code, logs, raw outputs, scoring rubric, environment config, and failures.
